# 0006. 인기 순위 노출 기준 — 최소 2표 + 비속어 블록리스트

- Status: Accepted
- Date: 2026-07-17

## Context

홈 첫 화면의 "지금 인기있는 선택" 섹션이 사용자 입력 원문을 무기준으로 노출했다.
`PopularityService` 의 점수는 `voteCount + recency 보너스(0~5)` 인데, 항목 수가
적은 초기 데이터에서는 **0표 항목이 recency 보너스만으로 랭킹에 진입**한다.
실제로 "ㅈㄴ맛있는거"(0표 비속어), "시간입력 0표" 같은 저품질 입력이 홈·인사이트
트렌드에 그대로 노출되어 첫인상 품질을 해쳤다. 인기 순위는 홈, `/insights/trends`,
`InsightsService` 의 top 항목 세 곳에서 소비된다.

## Decision

1. **집계 합산 후 `totalVotes >= 2` 임계.** "인기"는 여러 명이 동의한 항목이라는
   정의를 그대로 코드화한 것. 대소문자 동명 집계(1표+1표)는 합산 2표로 통과한다.
   `>= 1`(0표만 제외)은 1인 제안·셀프 투표 1회로 뚫려 기각.
2. **명백한 비속어 토큰 블록리스트.** 소문자화+공백 제거 후 containment 검사.
   토큰은 오탐(정상 상호명 차단) 위험이 없는 명백한 비속어로 최소 유지한다.
   형태소 분석·자모 변형 탐지 같은 본격 필터는 비용 대비 과잉으로 기각 —
   블록리스트는 best-effort 이며 우회 표기는 막지 못함을 수용한다.
3. **서비스 레이어 전역 적용** (`PopularityService` 의 두 집계 파이프라인).
   같은 "인기" 개념이 홈/트렌드/top 에서 다르게 동작하면 혼란이라 컨트롤러별
   분기는 기각. 세 소비처 모두 빈 결과 안전 확인: 홈 `th:if` 섹션 스킵,
   트렌드 empty-state, `InsightsService` top null 허용.

## Consequences

- 초기 데이터 규모에서는 홈 인기 섹션이 아예 비노출될 수 있다 — 의도된 동작
  (저품질 노출보다 미노출이 낫다).
- 셀프 투표 2회(2계정)로 임계를 뚫는 루프홀은 수용 — 표를 받은 비속어가
  실제로 관측되면 신고/차단 체계를 별도 ADR 로 다룬다.
- 블록리스트 토큰 추가는 이 ADR 의 범위 내 (기준 자체의 변경만 새 ADR).
- 회귀 가드: `PopularityServiceTest` (0·1표 제외, 집계 합산 통과, 블록리스트,
  정렬·limit·URL 병합 불변).
