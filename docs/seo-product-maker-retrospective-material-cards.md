# SEO 회고 글감 카드 — 프로덕트 메이커 관점 재구성용

> 이 문서는 원고가 아니다. 요즘IT 기고용 시리즈로 재편할 때를 위해, DateDate
> 프로젝트의 실제 커밋·코드·ADR·운영 사고에서 추출한 **회고 글감의 최소 단위**를
> 정리한 설계 문서다. 각 카드는 독립적으로 회고할 수 있고, 나중에 시리즈로
> 묶어 편성할 수 있도록 설계되어 있다.
>
> 기존 `seo-product-maker-retrospective-series.md` 1편은 *기술 How-to* 와
> *프로덕트 회고* 사이를 오가며 너무 많은 주제를 한 편에 압축했다. 본 문서는
> 그 압축을 풀어 **장면 단위로 다시 자른** 글감 카드 모음이다.
>
> 카드 작성 원칙:
> - 코드 설명이 아니라 *그 코드를 선택한 의사결정* 을 기록.
> - 모든 결정의 근거는 커밋/파일/ADR 로 검증 가능.
> - “효과가 좋았다” 같은 주관 단정 금지 — 검증되지 않은 수치는 *추가 확인 필요*.
> - 일반론 금지 — 이 프로젝트의 구체적 사건만.

---

## 사용 안내

- 카드 ID는 단순 일련번호다. **순서나 위계가 아니다.**
- 시리즈 6편 묶음 후보는 문서 하단의 [잠재 시리즈 묶음 후보](#잠재-시리즈-묶음-후보) 참고.
- 각 카드는 독립적으로 한 편의 글로 확장 가능하다. 두 카드를 합쳐 한 편이
  될 수도 있고, 한 카드가 두 편으로 쪼개질 수도 있다.
- 카드 안의 *글로 확장할 수 있는 질문* 은 서론/본론/결론의 골격을 잡는 용도다.

## 카드 01. SEO 를 “검색 노출”이 아니라 “제품의 자기소개” 문제로 재정의한 순간

### 관련 커밋 / 파일
- 커밋 `cdb552e` (2025-12-14) — *feat: SEO 적용 #9*
- 파일: `src/main/java/.../common/presentation/dto/SeoMetadata.java`
- 파일: `src/main/java/.../datedate/application/service/SeoService.java`
- 도메인 등록 후 약 한 달 — GSC 색인 0 으로 멈춤 상태.

### 당시 제품 문제
- MVP 를 만들어 친구에게 링크를 보냈지만, 도메인 이름으로 검색해도 결과가 비어
  있었다. 한 달이 지나도 색인이 1 로 올라가지 않았다.
- *기능* 은 동작했지만, *외부 세계가 이 사이트의 존재를 모른다* 는 사실은
  기능 개발만으로는 해결되지 않았다.

### 선택한 해결 방향
- SEO 작업의 첫 커밋을 단순한 메타 태그 추가가 아니라, **SeoMetadata 라는 record
  를 SSOT 로 두는 아키텍처 결정** 으로 시작했다.
- 동시에 robots, sitemap, manifest, favicon 의 *명함 노출 경로* 를
  `StaticResourceController` 로 명시했다.

### 대안과 포기한 것
- 각 컨트롤러에 메타 태그를 인라인으로 작성하는 안. 처음에는 빠르지만,
  hreflang/og:locale 같은 후속 결정이 늘어나면 모든 컨트롤러를 수정해야 했다.
- *키워드 위주의 SEO 작업* — 페이지가 무엇인지를 검색엔진에 알리기보다 키워드를
  채우는 방식. 정책 변화에 취약하다고 보고 포기.

### 프로덕트 메이커 관점의 고민
- 사이드 프로젝트라는 *내 정의* 와 외부 세계가 보는 *제품의 정의* 사이의 갭.
- *발견 가능성(discoverability)* 은 자동으로 따라오는 부산물이 아니라,
  의도적으로 설계해야 하는 제품 표면이라는 인식.
- 처음 한 줄의 코드가 6개월 뒤의 회귀 가능성을 결정한다는 관점.

### 실제 구현에서 얻은 인사이트
- 출시 첫 커밋부터 SEO SSOT 를 둔 결정은, 6개월 뒤 다국어/AdSense/og:locale
  같은 모든 결정을 *한 클래스 수정* 으로 흡수하게 만들었다 (카드 02 참조).
- *기능을 만든다* 와 *제품을 외부로 노출한다* 는 별도 작업이다.

### 글로 확장할 수 있는 질문
- 작은 서비스는 언제부터 SEO 를 고민해야 하는가?
- “MVP 가 동작한다”와 “MVP 가 외부 세계에 존재한다” 의 차이는 무엇인가?
- 검색엔진이 모르는 서비스는 존재한다고 말할 수 있는가?
- 개인 프로젝트라는 자기 정의는 누구를 위한 정의인가?

### 사용할 수 있는 근거 자료
- 커밋 `cdb552e` 의 변경 파일 17개.
- `docs/adr/common/seo/0001-seo-metadata-as-ssot.md` (Accepted 상태의 ADR).
- 회고 문서 `seo-product-maker-retrospective-series.md` §1.

### 원고화 시 주의점
- “이 결정 덕분에 트래픽이 N% 늘었다” 같은 수치 단정 금지 — *추가 확인 필요*.
- “SSOT 가 좋다” 의 일반론으로 흐르기 쉬움. *이 프로젝트에서 어떤 후속 결정이
  실제로 흡수되었는가* 를 카드 02 와 묶어 구체적으로 보여줘야 한다.

## 카드 02. “출시 첫 커밋부터 SeoMetadata SSOT 를 둬라” — 6개월 뒤가 증명한 결정

### 관련 커밋 / 파일
- `cdb552e` (2025-12-14) — record 첫 도입
- `6535183` (2026-04-07) — `adsEnabled` 필드 추가 (광고 정책 분기 흡수)
- `c8b970d` (2026-04-28) — `canonicalKo`, `canonicalEn`, `hreflangEnabled`,
  `ogLocale` 5 필드 한꺼번에 추가, 1612+/873− 라인
- `5e1e14e` (2026-05-09) — `og:locale:alternate`, `naverSiteVerification` 흡수
- 파일: `SeoMetadata.java`, `SeoService.java`

### 당시 제품 문제
- 출시 시점에 다국어/AdSense/JSON-LD/네이버 검증 같은 *미래 결정* 은 구체화되지
  않았다. 그러나 각 결정은 *모든 페이지의 메타* 에 동시에 영향을 미친다.
- 결정이 늘 때마다 모든 컨트롤러/템플릿을 손대는 *대수술* 을 막아야 했다.

### 선택한 해결 방향
- SEO 메타를 record + 서비스 한 곳에 집중. 컨트롤러는 `seoService.getXxxSeo()`
  호출만, 템플릿은 `seo.*` 렌더링만.
- 처음부터 `canonicalKo`, `canonicalEn`, `hreflangEnabled` 같은 *아직 안 쓸 필드*
  를 record 에 넣어둠.

### 대안과 포기한 것
- 컨트롤러 인라인 메타 — 빠르지만 회귀 위험.
- Thymeleaf fragment 안의 분기 로직만으로 처리 — 표현 계층에 분기 누적되어
  복잡도 폭발.
- “필요할 때 필드 추가” — 후속 대수술 비용을 과소평가.

### 프로덕트 메이커 관점의 고민
- *지금은 필요 없지만 6개월 뒤에 추가될 것이 거의 확실한 필드* 를 미리 두는
  결정의 비용과 가치. 과한 엔지니어링과 선견지명의 경계는 어디인가.
- 이 record 가 *기능 제품* 이 아니라 *외부 세계의 표현* 을 담는다는 의미에서,
  도메인 모델보다 더 안정적으로 설계할 이유.

### 실제 구현에서 얻은 인사이트
- AdSense 거절(카드 11), 영문 SEO 통합(카드 06), og:locale:alternate(카드 14)
  세 가지 결정이 모두 **컨트롤러 수정 0** 으로 흡수됐다.
- 일반화: *외부 세계에 노출되는 데이터 구조* 는 도메인 모델보다 *변하는 축* 이
  더 많다 — record 같은 불변/확장 친화적 타입이 잘 맞는다.

### 글로 확장할 수 있는 질문
- 처음부터 “쓰지 않을 필드”를 두는 것은 과한 엔지니어링인가?
- SSOT 가 *기능 도메인* 과 *외부 표현 도메인* 양쪽에 똑같이 가치가 있는가?
- record 의 불변성은 SEO 의 *결정성(deterministic)* 과 어떻게 맞물리는가?

### 사용할 수 있는 근거 자료
- `SeoMetadata.java` 의 필드 추가 히스토리 (`git log -p SeoMetadata.java`).
- `docs/adr/common/seo/0001-seo-metadata-as-ssot.md` 의 Consequences 절.

### 원고화 시 주의점
- 만능 패턴인 것처럼 일반화 금지. *작은 코드베이스* 라서 가능한 결정.
- “이 record 가 1612 줄 PR 을 막아줬다” 도 약간 과장. 실제로는 *어떤 부분이
  컨트롤러 수정 없이 흡수되었는지* 구체적으로 짚어야 한다.

### 당시 제품 문제
- “기능을 더 넣자”와 “외부 신호를 정돈하자” 사이의 시간 배분.
- 사이드 프로젝트 한 사람의 시간 100% 가 기능에 가지 않고 *제품 운영* 으로
  대거 흘러간 결정의 정당성.

- 약 4개월간 *SEO/콘텐츠/i18n/광고 정책* 회복을 명시적 우선순위로 두고,

- 이 결정이 만든 자산: ADR 10건, 플레이북 542줄, 회귀 테스트 인프라.

### 대안과 포기한 것
- *기능 우선* 방침으로 SEO 는 사용자 반응이 보일 때 하기. 발견 자체가 안 되니
  사용자 반응이 누적되지 않아 의사결정 불가.
- 외부 컨설팅/유료 도구. 비용·학습 곡선 모두 개인 프로젝트 규모 초과.

### 프로덕트 메이커 관점의 고민
- *내가 만든 것을 내가 어떻게 정의하느냐* 와 *외부가 그것을 어떻게 대하느냐*
  가 어긋날 때, 어떤 쪽을 정정해야 하는가.
- 사이드 프로젝트가 사이드 프로젝트라는 자기 정의 안에 머무를 권리는 있지만,
  배포된 순간 *제품을 운영하는 책임* 도 함께 발생한다.

### 실제 구현에서 얻은 인사이트
- 운영형 작업(SEO, i18n, 정책 페이지) 은 *기능* 으로 카운트되기 어려워서 항상
  뒤로 밀린다. 이걸 명시적 “우선순위 슬롯” 에 넣는 결정이 필요하다.
- 개인 프로젝트의 *운영 자산화* (ADR/플레이북/회귀 테스트) 는 사용자 0 명이라도
  *다음 프로젝트* 의 자산이 된다.

### 글로 확장할 수 있는 질문
- 사이드 프로젝트는 어디서부터 *제품* 이 되는가?
- 개인 프로젝트의 운영 작업은 어떤 기준으로 정당화되는가?
- 사용자 0 명인 프로젝트의 SEO 는 의미가 있는가?

### 사용할 수 있는 근거 자료
- `git log --since=2026-01 --until=2026-05 --oneline` 로 SEO/i18n 커밋과 기능
  커밋의 비율 분석.
- `docs/seo-evolution-playbook.md` 의 16 단계 타임라인.

### 원고화 시 주의점
- “이게 정답이다” 어조 금지. 다른 사이드 프로젝트는 다른 우선순위가 맞을 수
  있다.
- *시간 투입 대비 회수* 를 검증되지 않은 수치로 말하지 말 것.

## 카드 04. Sitemap 의 lastmod 를 “정직한 신호”로 만든 4단계 진화

### 관련 커밋 / 파일
- v1: `cdb552e` — 정적 `sitemap.xml` 1줄
- v2: `7eda6e2` (2026-01-06) — 정적 → `SitemapService` 동적 전환 (한때
  `LocalDate.now()` 가 들어가 있던 시기)
- v3: `65a8ff1` (2026-04-16) — `LASTMOD` 상수 도입 (수동 갱신)
- v4: `75fd434` (2026-05-08) — `BuildProperties#getTime()` + 도메인
  `findLatestActivity()` + ISO 8601 KST + `escapeXml()`
- 파일: `SitemapService.java`, `SitemapEntry.java`, `build.gradle (springBoot
  buildInfo)`

### 당시 제품 문제
- 검색엔진에 “이 페이지는 최근에 바뀌었어요” 라는 신호가 *진짜* 여야 한다.
- 초기 v2 에서 `LocalDate.now()` 를 박았더니 *재배포할 때마다 모든 페이지의
  lastmod 이 오늘로 갱신* 됨. Google 은 2023년부터 신뢰할 수 없는 lastmod 를
  점진적으로 무시한다.

### 선택한 해결 방향
- 정적 페이지: 빌드 시각 (`BuildProperties.getTime()`).
- 데이터 의존 페이지 (인사이트): 도메인 리포지토리의 `findLatestActivity()`.
- 출력 포맷: ISO 8601 풀 정밀도 + KST 오프셋. XML 5개 미리정의 엔티티 escape.

### 대안과 포기한 것
- `LASTMOD` 상수 유지 (v3) — 콘텐츠 변경 시마다 손 갱신, 망각 위험 커서 포기.
- 페이지별 콘텐츠 해시 추적 — 인프라 비용 과다.
- `LocalDate.now()` 유지 — *단기 효과는 같아 보이지만 장기 신뢰도를 깎는다*
  는 결론.

### 프로덕트 메이커 관점의 고민
- *효과가 좋아 보이는 신호* 를 만드는 것과 *정직한 신호* 를 만드는 것의 차이.
- “검색엔진을 자주 부르려고” lastmod 을 자주 갱신하면 *역효과* 라는 점은
  단순 트래픽 지표만 보면 알 수 없다. 정책 문서를 읽고 추론해야 한다.
- *결정성(deterministic)* — 같은 입력에 같은 출력을 내는 것 — 이 신호의 본질.

### 실제 구현에서 얻은 인사이트
- 외부 시스템과의 신호 교환에서 *진실성* 은 단기 최적화로 살 수 없다.
- 정직한 lastmod 을 위해 `BuildProperties` 의존이 추가됐다. 이런 *외부 신호
  품질을 위한 코드 의존성* 은 도메인 의존성과는 다른 종류의 비용으로 회계해야
  한다.

### 글로 확장할 수 있는 질문
- 외부 시스템에 보내는 신호의 “정직함”은 어떻게 정의되는가?
- 결정적(deterministic) 출력은 왜 SEO 의 핵심인가?
- 단기 효과와 장기 신뢰도가 충돌할 때 무엇을 우선해야 하는가?

### 사용할 수 있는 근거 자료
- `docs/adr/common/seo/0003-trustworthy-sitemap-lastmod.md`.
- `SitemapServiceHreflangTest.java` 의 결정성/ISO 형식/escape 검증.
- 커밋 `75fd434` 의 diff (LocalDate → OffsetDateTime 전환).

### 원고화 시 주의점
- “Google 정책” 인용은 출처 명시 (Google Search Central 블로그 2023 lastmod 글).
- v1~v4 진화를 *기술 스택 업그레이드* 처럼 자랑하지 말 것. 매 단계가 *그 시점의
  최선* 이었다는 톤이 더 정직하다.

## 카드 05. robots.txt `Disallow:/*/*` 가 정상 페이지까지 막아버린 사고

### 관련 커밋 / 파일
- 발생: `cdb552e` 시점의 robots.txt
- 회복: `69b9919` (2026-04-07) — `Disallow:/*/*` 제거, sitemap 6→11
- 진화: `3d2a37a` (2026-04-17) — `/*/2024/`~`/*/2028/` 연도 enumerate
- 확장: `5e1e14e` (2026-05-09) — `/*/2024/`~`/*/2035/` 11년치
- 파일: `src/main/resources/static/robots.txt`

### 당시 제품 문제
- UGC 일정 페이지 (`/{ownerId}/{year}/{month}`) 가 색인되면 사적 정보 노출과
  중복 콘텐츠 문제가 동시 발생.
- 이걸 막으려 `Disallow:/*/*` 를 넣었더니 `/runners/runs`, `/use-cases/*` 같은
  *정상 2-depth 페이지* 까지 차단되어 GSC 색인 실패.

### 선택한 해결 방향
- 광범위 패턴 → 연도별 명시 enumerate (`/*/2024/` ~ `/*/2035/`).
- 동시에 페이지별 `noindex` 메타로 이중 방어 (depth defense).

### 대안과 포기한 것
- robots.txt character class (`[0-9]`) 사용 — 표준 미지원이라 불가.
- robots.txt 제거하고 페이지별 `X-Robots-Tag` 헤더만 사용 — 가장 안전하지만
  컨트롤러 응답마다 신경 써야 함, 누락 위험.
- *경로 prefix 변경* (`/owner/{id}/...` 로 분리) — 라우팅 마이그레이션 비용 큼.

### 프로덕트 메이커 관점의 고민
- *방어선을 한 곳에 두지 않는다* — robots.txt 와 메타 robots 의 책임 분담.
- 도메인 자랄수록 *광범위 차단* 은 시한폭탄. 처음부터 *명시 차단* 의 비용을
  감수하는 것이 운영 친화적.
- 2035 까지의 enumerate 는 *현재 사람* 에게는 길어 보이지만 *미래 운영자* 에게는
  안전한 선택.

### 실제 구현에서 얻은 인사이트
- robots.txt 의 와일드카드 한계는 표준 자체의 제약이다 — 코드 우회로 푸는
  것이 더 안전.
- *“간결한 한 줄 차단”* 의 유혹을 거부하는 것이 도메인 멀티화의 첫걸음.

### 글로 확장할 수 있는 질문
- 광범위한 한 줄 정책과 명시적 다중 라인 정책 중 무엇이 운영에 좋은가?
- *암묵적 가정* (모든 2-depth URL 은 UGC) 이 깨질 때 어떻게 대처하는가?
- robots.txt 와 noindex 메타는 책임을 어떻게 나누어야 하는가?

### 사용할 수 있는 근거 자료
- `docs/adr/common/seo/0005-robots-disallow-narrowing.md`.
- 커밋 `69b9919` 의 robots.txt diff.
- GSC URL 검사 도구의 “robots.txt 에 의해 차단됨” 경고 (스크린샷이 있다면 인용).

### 원고화 시 주의점
- GSC 캡처 화면을 인용할 때 사용자 ID/도메인 비식별화 필요.
- “이렇게 하면 됩니다” 보다 *나는 이렇게 빠졌다* 톤이 글의 본질.

## 카드 06. 사이트맵과 robots 의 “지도/출입 정책” 비유와 신뢰의 단계 모델

### 관련 커밋 / 파일
- 커밋 흐름: `cdb552e` (정적 1줄) → `7eda6e2` (동적) → `75fd434` (신뢰도)
  → `c8b970d` (bilingual)
- 문서: `docs/seo-evolution-playbook.md` 의 *사이트맵 성숙도 모델* (L0~L5)

### 당시 제품 문제
- 사이트맵은 점점 더 많은 의미를 담는다 — *URL 목록* 에서 *변경 신호* 로,
  *언어 변종 매핑* 으로, *카테고리 분리* 로. 각 단계마다 *지금 어느 단계인지*
  알지 못하면 다음 결정을 하기 어렵다.

### 선택한 해결 방향
- 사이트맵을 *성숙도 단계 모델* 로 정리.
  - L0: 정적 1줄 — 존재 자체.
  - L1: 동적, 모든 URL.
  - L2: 정확한 lastmod.
  - L3: bilingual `<url>` + `xhtml:link`.
  - L4: 결정성 + 신뢰 가능한 lastmod (현재).
  - L5: sitemap-index, image/video 분리 (미래, URL 1000+).
- robots.txt 는 *출입 정책서* — 어떤 영역은 색인, 어떤 영역은 접근만, 어떤
  영역은 봇 자체 거부.

### 대안과 포기한 것
- “필요해질 때 단계 올리기” — 정확히 어디서 막혔는지 알기 어려움.
- 모든 단계를 처음부터 가는 것 — 작은 프로젝트에는 과한 투자.

### 프로덕트 메이커 관점의 고민
- 모든 인프라를 *성숙도 모델* 로 사고하면 *지금 더 투자할 가치가 있는가* 판단이
  쉬워진다.
- 비유 (지도 vs 출입 정책) 가 *비전공자 이해관계자* 와 의사소통할 때 코드보다
  강력하다.

### 실제 구현에서 얻은 인사이트
- *“지도와 출입 정책”* 비유는 디자이너/마케터에게 SEO 의사결정을 설명할 때
  실제로 잘 통한다 — 글에서도 그대로 사용 가능.
- 성숙도 모델은 *다음 프로젝트의 의사결정 비용* 을 줄인다.

### 글로 확장할 수 있는 질문
- 사이트맵은 어디까지 진화해야 하는가? 어디서 멈춰도 좋은가?
- robots.txt 는 차단보다 *신호* 에 가까운가?
- 작은 서비스의 *지도* 는 어떤 모양이어야 하는가?

### 사용할 수 있는 근거 자료
- `docs/seo-evolution-playbook.md` 의 사이트맵 성숙도 모델 절.
- 커밋 흐름 4개 비교.

### 원고화 시 주의점
- 단계 모델을 *모든 사이드 프로젝트에 적용 가능한 일반화* 로 포장 금지.
- L5 같은 미래 단계를 *예언* 하지 말 것 — 추가 확인 필요.

- 사용자에게는 잘 보이는데 *플랫폼* 에게는 안 보일 수 있다 — 이중 검증 필요.

### 실제 구현에서 얻은 인사이트
- 검증 파일은 *Content-Type* 까지 명시적으로 제어해야 한다.
- 이 패턴이 이후 `naver-site-verification` 파일에서도 동일하게 적용됨 (카드 18).

### 글로 확장할 수 있는 질문
- 사용자가 보는 것과 플랫폼이 보는 것의 차이는 어디에서 발생하는가?
- 정적 자산도 *컨트롤러로 노출* 해야 할 때는 언제인가?
- 검증 실패의 *침묵* 을 어떻게 감지할 것인가?

## 카드 08. 영문화 계획이 없을 때 미리 `canonicalKo/canonicalEn` 필드를 둔 이유

### 관련 커밋 / 파일
- `cdb552e` (2025-12-14) — `SeoMetadata` 첫 도입 시점에 이미 다국어 분리 필드 존재
- `c8b970d` (2026-04-28) — 4개월 뒤 실제 영문 SEO 통합. 컨트롤러 수정 거의 없음.

### 당시 제품 문제
- 다국어 출시는 *예정* 일 뿐 *확정* 이 아니었다. 그러나 그때 가서 `canonical`
  한 필드만 가진 record 를 확장하면 모든 컨트롤러를 다시 손대야 한다.

### 선택한 해결 방향
- `canonical`, `canonicalKo`, `canonicalEn`, `hreflangEnabled` 4 필드를
  *첫 커밋부터* record 에 둠. 컨트롤러는 `null` 또는 한 언어만 채울 수 있게.

### 대안과 포기한 것
- “다국어가 확정되면 그때 record 확장” — 비용을 *미래의 나* 에게 떠넘기는 선택.
- 단일 `canonical` + 별도 i18n 서비스 — 두 진실 공급원이 되어 동기화 위험.

### 프로덕트 메이커 관점의 고민
- *언제 미래의 결정을 데이터 모델에 미리 반영할 것인가* — 일반적으로는
  YAGNI 원칙이지만, *외부 표현 도메인* 은 변하는 축이 더 많고 비용 비대칭이
  크다.
- 영문화는 *기능* 이 아니라 *유입 전략* 이므로, 데이터 구조 결정의 비용/효용을
  다르게 회계해야 한다.

### 실제 구현에서 얻은 인사이트
- 4개월 후 `c8b970d` 가 컨트롤러 수정 없이 1612 줄을 흡수할 수 있었던 결정적
  이유는 *처음부터 이 필드가 있었다는 사실* 이다.
- 일반화 가능 통찰: *나중에 추가할 가능성이 높은 외부 표현 필드는 미리 둬도
  과한 엔지니어링이 아니다.*

### 글로 확장할 수 있는 질문

- “YAGNI” 원칙과 “외부 표현 도메인의 선반영” 은 어떻게 화해하는가?
- 영문화를 *코드 작업* 으로만 보지 않을 때 어떤 의사결정이 달라지는가?

### 원고화 시 주의점

## 카드 09. UGC noindex 페이지에 hreflang 을 발행하면 안 되는 이유 — 함정의 발견

### 관련 커밋 / 파일
- `c8b970d` — `hreflangEnabled` 플래그 도입
- 파일: `src/main/resources/templates/fragments/head.html` — `th:if="${seo.hreflangEnabled()}"`
- 회귀: `SeoServiceI18nTest` 의 dashboard/schedule SEO 가 `hreflangEnabled=false` 검증

### 당시 제품 문제
- UGC 일정 페이지(`/{ownerId}/2026/05`)는 *noindex* 해야 한다 (사적 정보 + 중복
  콘텐츠).
- 동시에 hreflang 을 발행하면 Google 이 *alt 그룹 전체의 인식을 포기* 한다. 즉,
  공개 페이지의 hreflang 효과까지 약화시킬 수 있다.

### 선택한 해결 방향
- `hreflangEnabled` 플래그를 record 에 두고, UGC/대시보드/관리자 페이지는
  `false`. 템플릿에서 `th:if` 로 hreflang/og:locale:alternate 발행 가드.

### 대안과 포기한 것
- *모든 페이지에 hreflang 발행* — Google 인식 포기 위험.
- 페이지별로 템플릿을 분기 — 컨트롤러/템플릿 수 폭증.

### 프로덕트 메이커 관점의 고민
- *공개 vs 비공개* 의 경계가 코드 레벨에서 *플래그 하나* 로 표현되는 것이
  운영적으로 안전하다.
- 검색엔진의 *알트 그룹 인식 포기* 같은 함정은 공식 문서를 정독하지 않으면
  보이지 않는다. *기능을 만드는 시간* 외에 *외부 시스템의 룰을 읽는 시간* 이
  별도 예산.

### 실제 구현에서 얻은 인사이트
- *공식 문서를 *코드 변경 전에* 직독* 하는 시간을 예산화해야 한다.
- 외부 시스템과의 약속은 *기본값* 이 위험 — 명시적 옵트인/옵트아웃 플래그가
  안전.

### 글로 확장할 수 있는 질문
- 외부 시스템(검색엔진)의 함정은 어떻게 발견하는가?
- 공식 문서 직독 vs 시행착오 — 무엇이 더 빠른가?
- “기본값으로도 동작” 과 “명시적 플래그” 는 어떤 비용 구조인가?

### 사용할 수 있는 근거 자료
- `head.html` 의 `th:if="${seo.hreflangEnabled()}"` 가드.
- `SeoServiceI18nTest` 의 dashboard noindex 케이스.
- `docs/adr/common/seo/0004-hreflang-canonical-locale-toggle.md`.

### 원고화 시 주의점

  조정.

## 카드 10. 언어 토글 KO=nofollow, EN=alternate hreflang 의 비대칭 설계

### 관련 커밋 / 파일
- `5e1e14e` (2026-05-09) — `header.html` 의 토글 rel 속성 비대칭 적용
- 파일: `src/main/resources/templates/fragments/header.html`

### 당시 제품 문제
- 언어 토글 버튼이 검색엔진에는 *별도 페이지로 색인 시도* 됨. `?lang=ko` 와
  *기본 페이지* 가 같은 내용인데 별도 URL 처럼 보일 수 있음.

### 선택한 해결 방향
- KO 토글: `rel="nofollow"` — 기본 페이지와 동일한 콘텐츠이므로 색인 대상에서
  자기 자신을 제외.
- EN 토글: `rel="alternate" hreflang="en"` — 영문 색인 보조 신호.

### 대안과 포기한 것
- 양 토글 모두 `alternate` — KO 가 canonical 과 동일한데 alternate 부여 시
  *자기 자신 alternate* 라는 의미 충돌.
- 양 토글 모두 `nofollow` — 영문 색인 보조 신호 손실.
- 라우팅을 `/ko`, `/en` 경로 기반으로 마이그레이션 — 전체 컨트롤러 변경 비용.

### 프로덕트 메이커 관점의 고민
- *기본값 언어* 와 *대안 언어* 사이의 신호 비대칭. canonical 정책의 미세한
  차이가 UI 의 attribute 한 글자로 표현된다는 사실.
- “양쪽이 평등” 한 토글이 사용자에게는 자연스럽지만, 검색엔진에는 비대칭이
  더 정확한 신호.

### 실제 구현에서 얻은 인사이트
- *사용자 경험의 대칭* 과 *검색엔진 신호의 비대칭* 이 충돌해도 OK — 둘은 다른
  레이어다.
- 작은 attribute 하나가 *알트 페이지 노이즈* 를 차단한다.

### 글로 확장할 수 있는 질문
- 사용자에게 평등해 보이는 UI 가 검색엔진에는 평등해야 하는가?
- canonical 의 “기본” 언어는 어떻게 정해지는가? 언어인가, 시장인가?
- *기능 평등* 과 *신호 비대칭* 은 어떻게 화해하는가?

### 사용할 수 있는 근거 자료
- `header.html` 토글 부분의 diff.
- `docs/adr/common/seo/0004-hreflang-canonical-locale-toggle.md` 의 Decision 절.

### 원고화 시 주의점
- “이렇게 하면 영문 색인이 됩니다” 단정 금지. 효과는 *추가 확인 필요*.

## 카드 11. 언어 토글이 “한 번에 안 먹는 버그” — 사용자 의도와 시스템 상태의 동기화

### 관련 커밋 / 파일
- 회복: `CookieThenAcceptLanguageLocaleResolver` 의 `setLocale` 에서 request
  attribute 캐시 도입
- 파일: `src/main/java/.../common/infrastructure/config/CookieThenAcceptLanguageLocaleResolver.java`
- 테스트: `CookieThenAcceptLanguageLocaleResolverTest`
- ADR: `docs/adr/common/i18n/0002-immediate-locale-application.md`

### 당시 제품 문제
- 사용자가 KO → EN 토글을 클릭해도 *그 요청의 응답* 에는 영문이 안 나옴. 새로고침
  한 번 더 해야 적용. 사용자 신뢰 손상.

### 선택한 해결 방향
- `setLocale` 에서 `request.setAttribute(...)` 로 캐시 + 쿠키 응답. `resolveLocale`
  은 0번 분기에서 attribute 를 먼저 보도록 변경.

### 대안과 포기한 것
- 토글 클릭 시 강제 redirect — URL 깜빡임 + history 두 단계.
- 클라이언트 사이드 i18n — 서버 렌더링과 이중 관리.

### 프로덕트 메이커 관점의 고민
- 사용자가 *명시적으로 선택한 의도* 가 같은 요청 안에서 즉시 반영되지 않을 때
  *신뢰* 가 깎인다. 한 번의 토글이 한 번의 시각적 변화로 끝나야 한다.
- 기술적으로 정상 동작(쿠키 응답 시점)이라도 *사용자 기대* 와 어긋나면 버그.

### 실제 구현에서 얻은 인사이트
- *동기적 상태* 와 *비동기적 영속* 을 분리해서 동시에 관리해야 한다 (request
  attribute + cookie).
- 다국어 지원의 핵심 이슈는 *번역 품질* 이 아니라 *상태 동기화* 일 수 있다.

### 글로 확장할 수 있는 질문
- 사용자 의도와 시스템 상태가 같은 요청 안에서 어긋나면 누가 옳은가?
- “기술적으로 정상 동작” 과 “사용자 경험상 정상” 의 차이는 어디인가?
- 다국어는 번역인가, 상태 관리인가?

### 사용할 수 있는 근거 자료
- `CookieThenAcceptLanguageLocaleResolver.java` 의 `resolveLocale` / `setLocale`
  메서드.
- `docs/adr/common/i18n/0002-immediate-locale-application.md`.

### 원고화 시 주의점
- Spring 내부 동작에 너무 깊이 들어가지 말 것. *문제의 본질* 은 동기화이지
  Spring 의 디테일이 아니다.

## 카드 13. URL 전략의 트레이드오프 — 쿼리 vs 경로 vs 서브도메인

### 관련 커밋 / 파일
- `c8b970d` — `?lang=en` 쿼리 방식 채택
- 파일: `SeoService.java` 의 `canonicalEn(path)` 헬퍼
- 문서: `docs/seo-evolution-playbook.md` 의 다국어 URL 의사결정 트리

### 당시 제품 문제
- 영문 SEO 도입 시 URL 전략 결정 필요:
  - 쿼리 (`?lang=en`) — 라우팅 변경 최소
  - 경로 (`/en/...`) — 검색엔진 인식 명확
  - 서브도메인 (`en.example.com`) — 분리 인프라

### 선택한 해결 방향
- 쿼리 방식 채택. 기존 라우팅이 단일 path 에 묶여 있어 마이그레이션 비용 큼.
- *단점 인지* — Googlebot 이 lang 파라미터를 무시할 수 있음. 이를 *hreflang*
  과 *canonical* 의 명확한 신호로 보완.

### 대안과 포기한 것
- 경로 기반 — 컨트롤러/리다이렉트 전면 재설계.
- 서브도메인 — DNS/인증서/세션 분리 비용. 사이드 프로젝트 규모 초과.

### 프로덕트 메이커 관점의 고민
- *기술적으로 가장 깨끗한 선택* (경로 기반) 과 *현재 규모에 맞는 선택* (쿼리) 의
  거리.
- 마이그레이션 비용이 *내가 감수할 만한가* 의 질문 — 같은 결정도 팀이 커지면
  답이 다르다.

### 실제 구현에서 얻은 인사이트
- “베스트 프랙티스” 와 *현재 프로젝트의 베스트 선택* 은 다르다.
- 쿼리 방식의 단점은 *다른 신호* (hreflang/canonical) 로 보완 가능 — 단,
  그 보완 신호 자체가 정확해야 한다.

### 글로 확장할 수 있는 질문
- 베스트 프랙티스를 따르지 않을 권리는 누구에게 있는가?
- “규모에 맞는 선택” 은 어떻게 정의하는가?
- 마이그레이션 비용은 어떤 형태로 누적되는가?

### 사용할 수 있는 근거 자료
- `SeoService.canonicalEn()` 의 구현.
- `docs/seo-evolution-playbook.md` 의 URL 전략 의사결정 트리.

### 원고화 시 주의점
- “쿼리 방식이 좋다/나쁘다” 단정 금지. *우리 규모에 맞다* 가 맞는 톤.

### 실제 구현에서 얻은 인사이트

### 글로 확장할 수 있는 질문
- 공유 링크의 첫인상은 본문보다 중요한가?

## 카드 15. AdSense 거절을 *제품 가치 점검의 기회* 로 재해석한 순간

- 정책 문서 직독 (4개) + 코드/콘텐츠와 1:1 매핑 (감사 문서) 이 *광고 정책* 을
  *제품 의사결정* 으로 번역하는 길이었다.
- 거절 사유의 *문자 그대로* 가 아니라 *의도* 를 읽는 작업.

### 글로 확장할 수 있는 질문
- 광고 승인 실패는 무엇을 알려주는가?

- 광고 수익화 이전에 *콘텐츠 신뢰* 를 어떻게 측정 가능하게 만들 것인가?

### 사용할 수 있는 근거 자료

- 실시간 hook (페이지 변경 즉시 호출) — 빈번한 호출, 정책 위반 위험.
- IndexNow 미사용 — 검색엔진 발견 속도 의존.
- 실패 시 재시도 — 복잡도/디버깅 비용.

### 프로덕트 메이커 관점의 고민
- SEO 는 *비동기적 보상* — 실패가 즉각 서비스에 영향 주지 않게 설계하는 것이
  운영 안정성의 전제.
- *비밀번호 인증 같은 동기 작업* 과 *SEO 같은 비동기 신호* 는 다른 실패 정책이
  맞다.

### 실제 구현에서 얻은 인사이트
- *Fail-soft* 와 *fail-fast* 는 도메인에 따라 다르다 — SEO 는 fail-soft 가
  맞다.
- 환경변수 ON/OFF 토글은 *카나리* 같은 역할을 한다.

### 글로 확장할 수 있는 질문
- 어떤 작업이 fail-soft 가 맞고 어떤 작업이 fail-fast 가 맞는가?
- 비동기적 보상의 운영 정책은 어떻게 설계해야 하는가?
- IndexNow 같은 *플랫폼 통합* 은 실제로 효과가 있는가? (*추가 확인 필요*)

### 사용할 수 있는 근거 자료
- `IndexNowService.java` 의 예외 처리 흐름.
- `application.yaml` 의 indexnow 설정 키.

### 원고화 시 주의점
- IndexNow 자체의 효과 단정 금지 — *추가 확인 필요*.

## 카드 22. Cache-Control + Content-Language 차등 — “공개 SEO 페이지와 UGC 의 캐시 정책 분리”

### 관련 커밋 / 파일
- `34335db` (2026-04-14) — WebConfig 에 캐시 정책 도입
- 파일: `WebConfig.java` 의 `PUBLIC_SEO_PATHS`, `PUBLIC_SEO_PREFIXES` 와 캐시
  인터셉터

### 당시 제품 문제
- 공개 SEO 페이지는 *공유 캐시(CDN)* 활용으로 빠른 응답이 필요.
- UGC 페이지는 *사용자 개별* 콘텐츠 — 공유 캐시 금지.
- 두 정책을 한 곳에서 깔끔히 분리해야 함.

### 선택한 해결 방향
- 공개 SEO: `Cache-Control: public, max-age=3600, s-maxage=86400`,
  `Vary: Cookie, Accept-Language`.
- UGC: `Cache-Control: private` (또는 no-cache).
- `Content-Language` 헤더로 로케일 신호 명시.

### 대안과 포기한 것
- 전체 사이트 동일 캐시 정책 — UGC 노출 위험.
- nginx 단에서만 캐시 분기 — 인프라 환경 의존.

### 프로덕트 메이커 관점의 고민
- 캐시는 *성능* 의 도구이자 *프라이버시* 의 위험이다 — 두 축을 동시에 고려해야
  한다.
- *공개* 와 *개인* 의 경계가 *URL* 만으로 표현되지 않을 때 (같은 URL 이 로그인
  상태에 따라 다른 내용) Vary 헤더가 중요해진다.

### 실제 구현에서 얻은 인사이트
- 캐시 정책은 *SEO* 와 *프라이버시* 가 동시에 닿는 영역.
- 페이지 유형 매트릭스 (콘텐츠/UGC/관리) 가 캐시 정책에도 그대로 적용된다.

### 글로 확장할 수 있는 질문
- 캐시는 성능 문제인가, 프라이버시 문제인가?
- 같은 URL 이 사용자에 따라 다른 내용을 가질 때 어떻게 신호를 줘야 하는가?
- Vary 헤더는 SEO 의 어디에 영향을 주는가?

### 사용할 수 있는 근거 자료
- `WebConfig.java` 의 `PUBLIC_SEO_PATHS`, 인터셉터 로직.
- 커밋 `34335db` 의 diff.

### 원고화 시 주의점
- HTTP 캐시 디테일에 너무 깊이 들어가지 말 것. *제품 분류 → 캐시 분기* 의 흐름만
  보여주면 충분.

## 카드 23. JSON-LD 의 점진적 확장 — “스키마는 한 번에 다 넣지 않는다”

### 선택한 해결 방향
- 페이지 유형별로 *최소 의미* 의 스키마부터 시작 (WebApplication / WebSite).

### 대안과 포기한 것
- 한 번에 5~10 종 동시 적용 — 측정 불가, 잘못된 데이터 발견 어려움.
- 자동 스키마 생성 라이브러리 — 도메인 의존성 + 정확도 손해.

### 프로덕트 메이커 관점의 고민
- *과한 신호* 는 *부족한 신호* 만큼 위험할 수 있다 — Google 이 자기 모순 신호로
  볼 수 있음 (예: FAQ 페이지의 빈 Q&A).
- *측정 가능한 단위로 변화* 시키는 것이 의사결정의 기본.

### 실제 구현에서 얻은 인사이트
- JSON-LD 는 *콘텐츠와 동기화* 되어야 한다 — Use-case FAQ 가 q1 키 존재 여부로
  조건부 생성되는 이유.
- 스키마는 *콘텐츠의 거울* — 콘텐츠가 비어 있으면 스키마도 비워야 한다.

### 글로 확장할 수 있는 질문
- 스키마는 한 번에 다 넣을수록 좋은가?
- 잘못된 스키마는 어떤 *역효과* 를 내는가?
- 콘텐츠와 스키마의 동기화는 어떻게 가드하는가?

### 사용할 수 있는 근거 자료

- 커밋 `c7ddfe8`, `69b9919` 의 diff.

### 원고화 시 주의점

## 카드 26. 검색 결과 미리보기와 실제 페이지의 “기대값 정렬” — 작은 서비스의 신뢰 전략

### 프로덕트 메이커 관점의 고민

  쌓인다.
- 작은 서비스의 마케팅 카피라이팅은 *“정렬”* 의 다른 이름.

### 글로 확장할 수 있는 질문

- 검색 미리보기 ↔ 페이지 ↔ 기능 의 “정렬” 은 어떻게 측정하는가?

## 카드 27. 개발자 모자 ↔ 프로덕트 메이커 모자 — 같은 코드 위에서 두 시선의 전환

- *기능* 을 만드는 자아와 *제품* 을 운영하는 자아가 같은 사람일 때, 의사결정의
  기준이 흔들린다.
- “기능이 동작하면 됐지” 와 “검색엔진이 이걸 이해하나, 사용자가 신뢰하나, 봇이

### 선택한 해결 방향
- 의도적으로 *모자 전환* — 같은 코드를 검토할 때 다른 질문을 던짐.

### 대안과 포기한 것
- 한 자아만 — 기능 우선 시 SEO/UX 손실, 제품 우선 시 코드 품질 손실.
- 자아 분리 (역할 분담) — 사이드 프로젝트에선 불가능.

### 프로덕트 메이커 관점의 고민
- *같은 코드에 두 개의 시선* 을 *명시적으로* 분리하는 작업.
- *모자를 바꾸는 트리거* — PR 만들 때, 배포 직전, GSC 리뷰 시.

### 실제 구현에서 얻은 인사이트
- 두 모자 사이를 *의식적으로* 오갈 때 의사결정이 깊어진다.
- 동료 리뷰가 없는 사이드 프로젝트에서 *모자 전환* 은 일종의 *셀프 코드 리뷰*.

### 글로 확장할 수 있는 질문
- 한 사람이 두 역할을 동시에 할 때 어떤 신호로 전환하는가?
- 개발자의 질문과 프로덕트 메이커의 질문은 어떻게 다른가?
- 셀프 코드 리뷰의 구조는 어떻게 만들 수 있는가?

### 사용할 수 있는 근거 자료
- 6개월 커밋 흐름.
- 회고 문서 §7.

### 원고화 시 주의점
- *자기 계발서* 톤으로 흐르지 말 것. 항상 *코드/사고 사례* 와 묶어서.

## 카드 28. ADR 와 플레이북 — “미래의 나” 와 “다음 프로젝트” 를 위한 자산화

### 관련 커밋 / 파일
- `3f8621c` (2026-05-10) — *docs: SEO 진화 플레이북* (542줄)
- 디렉토리: `docs/adr/common/seo/` (7개), `docs/adr/common/i18n/` (3개)
- 문서: `docs/audit/adsense-low-value-content-policy-mapping.md`

### 당시 제품 문제
- 6개월 사이의 결정들을 *코드로만* 남기면 *왜* 가 빠르게 잊힌다.
- 다음 사이드 프로젝트를 시작할 때 같은 시행착오를 반복하기 쉽다.

### 선택한 해결 방향
- *ADR* — 모든 *결정* 은 별도 markdown 으로 (Status, Context, Decision,
  Alternatives, Consequences).
- *플레이북* — 6개월 흐름의 *서사적 통합* + 출시 전 체크리스트 + 안티패턴 10개.
- *Audit* — 외부 정책 문서와 코드/콘텐츠의 1:1 매핑.

### 대안과 포기한 것
- 모든 결정을 PR 설명에만 남기기 — 시간 흐름에 따라 찾기 어려움.
- 위키 (Notion 등) — 코드와 분리, 동기화 부담.
- 자산화 포기 — 다음 프로젝트의 시작 비용 증가.

### 프로덕트 메이커 관점의 고민
- *내가 만든 결정이 6개월 뒤의 나에게 친절한가* — 코드만으로는 답이 아니다.
- 사이드 프로젝트의 *진짜 자산* 은 코드가 아니라 *결정 기록* 일 수 있다.
- 자산화는 *이번 프로젝트* 의 추가 비용이지만 *다음 프로젝트* 의 가속.

### 실제 구현에서 얻은 인사이트
- ADR 은 *얇게* 자주 쓰는 것이 *두껍게* 가끔 쓰는 것보다 좋다.
- 플레이북 한 편이 *블로그 시리즈의 1차 자료* 로 그대로 활용 가능 (실제로 이
  문서의 입력).

### 글로 확장할 수 있는 질문
- 사이드 프로젝트의 자산은 코드인가, 결정 기록인가?
- ADR 은 얼마나 자주 써야 하는가?
- 다음 프로젝트의 시작 비용을 어떻게 낮출 것인가?

### 사용할 수 있는 근거 자료
- `docs/adr/` 의 디렉토리 구조.
- `docs/seo-evolution-playbook.md` 의 16 단계 타임라인.
- 본 문서 자체가 *플레이북의 재사용* 사례.

### 원고화 시 주의점
- ADR 의 형식적 측면만 자랑하지 말 것. *왜 쓰는가* 의 관점이 본질.

## 카드 30. www / non-www / HTTP / HTTPS 의 “리디렉션 체인 1회 이내” 운영 규율

### 관련 커밋 / 파일
- `19f9286` (2026-01-06) — HTTP/HTTPS 4곳 통일
- 문서: `docs/troubleshooting/google-search-console-redirect.md`
- ADR: `docs/adr/common/seo/0002-http-to-https-unification.md`

### 당시 제품 문제
- robots.txt / sitemap.xml / application.yaml / SeoService 의 baseUrl 이 4곳에서
  http / https 가 섞여 있어서 GSC *“리디렉션이 포함된 페이지”* 경고.

### 선택한 해결 방향
- 4곳을 *동시 커밋 1회* 로 모두 https 로 통일.
- `curl -IL` 로 리디렉션 횟수 1회 이내 검증을 운영 규율로.

### 대안과 포기한 것
- 부분 통일 — 회귀 위험.
- nginx 만 강제 — 코드 안의 http 노출.

### 프로덕트 메이커 관점의 고민
- *진실 공급원이 분산* 되어 있는 상태는 *동기 변경* 이 강제될 때 비용이 폭발.
- 운영 규율 (`curl -IL`) 을 *체크리스트* 로 명문화하는 것이 미래의 나에게 친절.

### 실제 구현에서 얻은 인사이트
- 작은 운영 규율 한 줄이 *반복 사고* 를 막는다.

### 글로 확장할 수 있는 질문
- 진실 공급원의 *분산* 은 어떻게 *모임* 되어야 하는가?
- *체크리스트* 는 어떻게 자산이 되는가?

### 사용할 수 있는 근거 자료
- `docs/troubleshooting/google-search-console-redirect.md`.
- 커밋 `19f9286` 의 diff.

# 핵심 메시지 후보

> 전체 시리즈를 관통할 수 있는 핵심 메시지의 후보. 한 편/한 단락의 닫는 문장으로
> 활용하거나, 시리즈 머리말의 thesis 로 변형 가능.

1. **SEO 는 검색엔진을 속이는 기술이 아니라, 제품의 의도를 검색엔진과 사용자에게
   *일관되게* 설명하는 일이다.** — 같은 의미를 메타·페이지·기능 세 표면에서
   어긋나지 않게 정렬하는 작업이 *작은 서비스의 신뢰 전략* 이다.

2. **작은 프로젝트의 SEO 는 트래픽 문제가 아니라 제품 *발견 가능성* 의 설계
   문제다.** — 발견되지 않는 제품은 존재한다고 말할 수 없고, 발견 가능성은
   기능을 만들면 자동으로 따라오는 부산물이 아니다.

3. **개발자는 검색 노출을 *구현* 하는 사람이 아니라, 발견 가능성을 *제품 구조에
   심는* 사람이다.** — SeoMetadata SSOT, 회귀 테스트, 페이지 유형 매트릭스는
   모두 *구조에 심는* 작업의 다른 이름이다.

4. **다국어는 번역의 문제가 아니라 정보 구조 재설계의 문제다.** — 영문화는 단어
   교체가 아니라 *canonical 정책 결정 시점* 부터 데이터 구조에 영입해야 하는
   일이다.

5. **AdSense 거절은 광고 승인 실패가 아니라 제품 가치 진단서다.** — 검색엔진과
   광고 플랫폼은 같은 신호를 본다: *“사용자가 머물 만한 콘텐츠가 있는가”*.
   거절을 진단서로 읽으면 *콘텐츠 신뢰의 측정 가능한 신호* 가 보인다.

6. **SEO 는 한 번 끝나는 작업이 아니라 *반복되는 운영* 이다.** — 회귀 테스트와
   ADR/플레이북은 *기능 개발 속도를 늦추지 않으면서 깨지지 않게 유지* 하기
   위한 인프라다.

7. **개인 프로젝트라는 자기 정의는 만든 사람의 것이지, 외부 세계의 것이 아니다.**
   — 배포된 순간 제품이 된다. 사이드 프로젝트의 *제품 운영 책임* 은 *사이드*
   라는 단어로 회피되지 않는다.

8. **외부 표현 도메인은 도메인 모델보다 변하는 축이 더 많다.** — 그래서 *지금은
   쓰지 않는 필드* (`canonicalEn`, `hreflangEnabled`, `adsEnabled`) 를 처음부터
   record 에 두는 것이 과한 엔지니어링이 아니다. *6개월 뒤의 나에게 친절한* 결정.

# 부록 B. 기존 회고 시리즈 1편의 비판적 평가 요약

> `docs/seo-product-maker-retrospective-series.md` 1편에 대한 비판적 검토. 본
> 카드 문서를 만든 *재구성의 이유*.

**잘된 부분**
- 출발이 *기획 의도* + *프로젝트 맥락* 으로 시작 — 기술 How-to 톤을 피하려는 의도
  자체는 명확.
- 참고 근거 표 (코드/문서/커밋 매핑) 가 검증 가능.

**다시 자를 여지가 큰 부분**
- §2 *처음 마주한 문제 5가지* 가 한 섹션에 압축 — 각각을 독립 카드로 풀면 더
  강한 회고가 됨. → 본 문서 카드 01, 11, 15, 17~18, 20 로 분산.
- §3 *Sitemap 진화* 가 *기술 단계* 처럼 흐름 — *4단계 진화 자체* 를 한 카드로
  하되, *왜 v2 가 그 시점엔 좋아 보였나* 의 깊이를 추가. → 카드 04.
- §4 *Canonical 과 hreflang* 가 *결과 설명* 위주 — *처음부터 필드를 분리한 선견*
  의 의사결정 스토리가 약함. → 카드 02, 08 로 분리.
- §5 *메타 태그/OG* 가 *베스트 프랙티스* 톤 — *기대값 정렬의 신뢰 전략* 으로
  재해석 가능. → 카드 26.
- §6 *회귀 테스트* 가 *불변 목록* 만 — 각 테스트가 *어떤 사고를 막는지* 의 사례
  부족. → 카드 12, 20.
- §7 *얻은 인사이트* 5개가 좋지만 *너무 마지막에 압축* — 시리즈 다른 편의
  *결론* 으로도 활용 가능. → 핵심 메시지 후보로 분산.
- §7 마지막 *두 모자 비유* 가 회고의 핵심인데 *맨 끝* 에 짧게 — 시리즈의
  *프롤로그* 또는 *에필로그* 로 확대. → 카드 27.

**현재 1편 구조의 한계**
- 1편 안에 *Sitemap·Robots·Canonical·hreflang·OG·회귀 테스트* 6 주제가 모두
  들어가 분량 압박. 각 주제의 *결정 스토리* 가 짧아짐.
- *기술* 과 *제품* 사이의 톤이 섹션마다 흔들림 — 카드 단위로 자르면 톤 유지가
  쉬워짐.
