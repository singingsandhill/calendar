```
당신은 시니어 백엔드 아키텍트, 시니어 애플리케이션 설계 리뷰어, 그리고 유지보수성/확장성/운영성을 중시하는 코드 분석 전문가입니다.

내 프로젝트는 `DateDate`라는 그룹 일정 조율 서비스입니다.

기술 스택과 구조는 다음과 같습니다.

- Spring Boot 4
- Java 21
- Thymeleaf 기반 SSR
- Vanilla JS
- CSS
- JPA/Hibernate
- 헥사고날 아키텍처 (presentation / application / domain / infrastructure)
- 핵심 도메인: Owner, Schedule, Participant
- 핵심 기능: 개인 owner 페이지, 월별 일정 생성, 참여자 추가, 날짜 선택, 저장, 공유 링크 기반 일정 조율

이제 이 프로젝트를 **UI/UX가 아니라 소스코드 구조와 아키텍처 관점에서 정밀 분석**해주세요.

중요한 것은 단순한 코드 스타일 리뷰가 아닙니다.
이 서비스가 앞으로 기능이 늘어나고 실제 운영이 길어질 때,
**어디서 구조적 부채가 생길지, 무엇을 먼저 정리해야 하는지**를 분석하는 것이 목적입니다.

# 1. 분석 목표

3. 날짜/시간/상태/권한/공유 같은 핵심 도메인 영역이 안정적으로 설계되어 있는지 점검
4. 단기 개선 포인트와 구조적 리팩토링 포인트를 분리
5. 실제 수정 가능한 수준으로 파일/레이어 기준 개선 방향을 제안

# 2. 프로젝트 특성을 반드시 반영할 것

이 프로젝트는 일반 게시판이 아니라 **날짜와 참여 상태를 중심으로 하는 일정 조율 도구**입니다.
따라서 다음 특성을 충분히 고려해주세요.

- 날짜/월/선택 상태가 핵심 데이터다
- 여러 참여자의 상태가 하나의 일정에 모인다
- 개인(owner) 페이지와 공유 접근 흐름이 함께 존재한다
- 추후 반복 일정, 시간대, 응답 상태 확장 가능성이 있다
- 공개 링크 / 비공개 관리 / SEO 가능 공개 페이지 같은 구조 분리가 중요할 수 있다
- 현재는 단순해 보여도 도메인 규칙이 빠르게 얽히기 쉬운 유형이다

즉, “일정 조율 서비스”라는 도메인 특성을 전제로 분석해주세요.

# 3. 우선 분석 관점

아래 관점을 반드시 포함해서 분석해주세요.

## 3-1. 도메인 모델링 관점
다음을 점검해주세요.

- Owner / Schedule / Participant의 책임이 명확히 분리되어 있는가
- 일정 조율 도메인 개념이 엔티티에 자연스럽게 반영되어 있는가
- 단순 CRUD 수준을 넘어 비즈니스 규칙이 어디에 위치하는가
- 날짜 선택, 참여자 제한, 중복 이름 방지, 월별 일정 중복 방지 같은 규칙이 적절한 레이어에 있는가
- 향후 반복 일정, 일정 상태, 확정일, 투표형 응답, 시간대 같은 기능이 추가될 때 현재 모델이 견딜 수 있는가

특히 아래를 봐주세요.
- Schedule이 단순 month container인지, 실제 aggregate root로 적절히 기능하는지
- Participant가 선택 상태를 직접 가지는 방식이 적절한지
- Owner 식별자와 공유 링크 모델이 확장 가능하게 설계되어 있는지

---

## 3-2. 날짜/시간 처리 관점
DateDate에서 매우 중요한 영역입니다. 반드시 깊게 봐주세요.

- 날짜 선택 로직이 어디에서 처리되는가
- day 값(1~31) 중심 처리 방식의 장단점
- 연/월/일 검증 로직의 위치와 일관성
- month 길이, 윤년, 월말 경계 처리 안정성
- `LocalDate`, `YearMonth` 등 날짜 타입 활용이 충분한가
- 문자열/숫자 기반 날짜 처리 때문에 향후 버그가 생길 가능성은 없는가
- 서버 렌더링과 JS 상호작용에서 날짜 표현이 일관적인가

특히, 현재 구조가 “월 단위 일정 선택 서비스”에 최적화된 것인지,
아니면 향후 시간 단위 일정으로 확장될 때 크게 깨지는 구조인지 평가해주세요.

---

## 3-3. 애그리거트/비즈니스 규칙 배치 관점
헥사고날 아키텍처를 실제로 잘 쓰고 있는지 보려는 목적입니다.

- 핵심 규칙이 domain에 있는가, service에 몰려 있는가, controller에서 새고 있는가
- application service가 orchestration만 하는지, 도메인 로직까지 과도하게 먹고 있는지
- validation이 presentation/application/domain 중 어디에 위치해야 적절한지
- JPA entity와 domain model이 분리되어 있는지, 아니면 사실상 persistence model에 종속되어 있는지
- repository port abstraction이 실질적인 의미가 있는지

특히 아래를 중점적으로 봐주세요.
- “참여자 최대 8명”
- “동일 이름 참여자 중복 방지”
- “해당 owner의 동일 연/월 schedule 중복 방지”
- “유효하지 않은 day 선택 방지”

이 규칙들이 현재 구조상 가장 적절한 자리에 있는지 판단해주세요.

---

## 3-4. 상태 관리 / 데이터 흐름 관점
이 프로젝트는 Thymeleaf SSR + Vanilla JS 구조입니다.
프론트 프레임워크가 없기 때문에 오히려 상태 관리가 암묵적으로 꼬일 수 있습니다.

다음을 점검해주세요.

- 서버 렌더링과 클라이언트 JS 상태가 어떻게 연결되는가
- 페이지 리로드 중심 흐름이 유지보수에 유리한지 불리한지
- 선택 중 상태(selectedDays), 선택된 참여자, 저장 후 갱신 흐름이 명확한가
- hidden input, dataset, 전역 변수, inline script 등에 상태가 흩어져 있지 않은가
- 향후 partial update/AJAX 강화 시 구조적으로 충돌하는 부분은 없는가

특히:
- 현재 구조가 “간단해서 괜찮은 것”인지
- 아니면 이미 UI 상태와 서버 상태가 은근히 엮이기 시작했는지
판단해주세요.

---

## 3-5. API 설계 관점
DateDate는 MVC + REST API가 공존하는 구조입니다.
다음 관점으로 API를 점검해주세요.

- API endpoint naming이 일관적인가
- owner 기준 API / schedule 기준 API / participant 기준 API가 자연스럽게 설계되어 있는가
- REST 자원 구조가 도메인에 맞는가
- GET/POST/PATCH/DELETE 사용이 적절한가
- 반환 DTO 구조가 프론트 요구에 과도하게 종속되어 있지 않은가
- 에러 코드 체계가 확장 가능하게 설계되어 있는가

추가로 다음을 봐주세요.

- MVC 화면 렌더링과 REST API 사이의 책임 경계가 분명한가
- 중복 로직이 controller 레벨에서 생기고 있지 않은가
- 추후 외부 클라이언트(앱, 별도 프론트엔드)가 붙기 쉬운 구조인가

---

## 3-6. 권한 / 공개 범위 / 식별자 관점
DateDate는 owner 기반 개인 페이지와 공유 링크 기반 참여가 존재합니다.
보안과 구조 양쪽 측면에서 점검해주세요.

- ownerId 기반 페이지 접근이 충분히 안전한가
- 고유 ID 정책이 단순해도 괜찮은 수준인지
- “소유자 관리 페이지”와 “참여자 접근 페이지” 구분이 명확한가
- 링크만 알면 가능한 동작 범위가 어디까지인지 코드상 명확한가
- 추후 비공개/공개/읽기전용/수정권한 분리 시 현재 구조가 확장 가능한가

그리고 다음 질문에 답해주세요.

- 현재 구조는 “간단한 개인용 링크 서비스”로는 충분한가?
- 아니면 권한 모델이 너무 암묵적이어서 나중에 큰 리팩토링이 필요한가?

---

## 3-7. 템플릿/프런트 구조 관점
UI 평가가 아니라 코드 구조 평가입니다.

- Thymeleaf template, fragments, static js/css 구조가 유지보수 가능하게 나뉘어 있는가
- 특정 페이지 템플릿이 너무 많은 책임을 갖고 있지 않은가
- DOM 조작 코드가 페이지마다 중복될 위험이 있는가
- 재사용 가능한 UI 조각과 페이지 전용 로직이 구분되어 있는가
- 점(dot) 렌더링, 참여자 드롭다운, 모달, 링크 복사 등 상호작용 코드가 어디까지 분리되어 있는가

가능하면 다음 분류로 진단해주세요.
- 템플릿 레벨 문제
- JS 구조 문제
- CSS 구조 문제
- SSR/JS 경계 문제

---

## 3-8. 테스트 가능성 관점
다음을 점검해주세요.

- 서비스 테스트가 핵심 비즈니스 규칙을 제대로 보호하는가
- 날짜 선택 로직이 단위 테스트하기 쉬운 구조인가
- participant/schedule 관련 제약 조건 테스트가 충분히 분리 가능한가
- MVC/API/controller 테스트와 domain/application 테스트의 역할 구분이 적절한가
- 테스트가 단순 happy path 위주인지, 경계값/예외 케이스도 보호하는지

특히 아래 케이스가 테스트 관점에서 잘 드러나는지 봐주세요.
- 2월 / 30일 / 31일
- 중복 참여자
- 최대 참여자 초과
- 이미 존재하는 연월 일정
- 존재하지 않는 owner / schedule / participant
- 잘못된 selection payload

## 3-9. 성능 및 쿼리 관점
현재는 규모가 작더라도 구조적으로 봐주세요.

- N+1 가능성이 있는가
- 달력 렌더링을 위해 필요한 데이터가 과하거나 부족하지 않은가

- 점 표시용 데이터 구조가 참여자 수/기능 확장 시 비효율적으로 바뀔 위험이 있는가

단기 성능 문제보다,
**기능 추가 후 성능 이슈가 터질 구조인지**를 더 중점적으로 평가해주세요.

# 5. 작업 방식

중요:
- 절대 추상적인 일반론만 말하지 마세요.
- 실제 코드베이스를 먼저 훑고, 그 구조를 기준으로 판단해주세요.
- “좋은 아키텍처 원론”을 나열하지 말고, DateDate에 실제로 해당하는 내용만 말해주세요.
- 과한 리팩토링 제안보다, 현재 프로젝트 규모에 맞는 현실적인 판단을 우선해주세요.
- 단기적으로 충분한 구조와, 장기적으로 위험한 구조를 구분해서 말해주세요.

작업 순서는 다음과 같이 진행해주세요.

1. DateDate 관련 패키지/템플릿/JS 구조 탐색
2. 핵심 흐름 추정 (owner 생성 → schedule 생성 → participant 추가 → selection 저장)
3. 비즈니스 규칙 위치 파악
4. 날짜/상태/권한/공유 구조 점검
5. 테스트 및 확장성 점검
6. 실제 리팩토링 우선순위 정리

최종 결과는 반드시
**“DateDate 코드 아키텍처 분석 보고서 + 실행 가능한 리팩토링 제안”**
형태로 작성해주세요.
```
