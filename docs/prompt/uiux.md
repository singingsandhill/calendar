```
당신은 시니어 프로덕트 디자이너, 시니어 프론트엔드 아키텍트, 그리고 SEO/수익화 최적화 경험이 있는 웹 서비스 리팩토링 전문가입니다.

내 프로젝트는 **DateDate**라는 캘린더/일정 조율 성격의 웹 서비스입니다.
목표는 단순히 예쁘게 바꾸는 것이 아니라, **사용자가 더 쉽게 이해하고 더 빠르게 행동할 수 있는 구조**로 화면 UI/UX를 전면 리팩토링하는 것입니다.

다음 원칙에 따라 작업해주세요.

# 1. 프로젝트 이해와 목표

이 서비스의 핵심 가치는 다음과 같습니다.

- 날짜/일정 정보를 빠르게 확인할 수 있어야 함
- 약속/일정 조율 또는 캘린더 기반 행동이 직관적이어야 함
- 첫 방문 사용자도 서비스 목적을 즉시 이해할 수 있어야 함
- 반복 사용자가 불편 없이 빠르게 핵심 기능에 진입할 수 있어야 함
- 추후 SEO 유입과 Adsense 수익화가 가능한 구조여야 함

즉, 이번 리팩토링은
**“캘린더 기능 중심 서비스” + “콘텐츠/랜딩/검색 유입 구조” + “광고 삽입 여지”**
를 동시에 만족해야 합니다.

# 2. 작업 목표

다음 6가지를 반드시 달성해주세요.

1. **정보구조(IA) 재정리**
   - 첫 화면에서 서비스 목적이 명확히 보이도록 개선
   - 사용자의 주요 행동 흐름(보기 / 생성 / 참여 / 공유)을 명확히 분리
   - 메뉴 구조와 페이지 역할을 재정의

2. **UI/UX 전면 리팩토링**
   - 시각적으로 더 현대적이고 명확하게 개선
   - 불필요한 장식보다 사용성 중심
   - 모바일/태블릿/데스크탑 반응형 최적화
   - 사용자 인지 부담을 줄이고 클릭 동선을 짧게 만들기

3. **SEO 친화 구조 반영**
   - 검색엔진이 이해하기 쉬운 semantic markup
   - 각 페이지의 title / meta description / heading hierarchy / canonical 전략 제안
   - indexable page와 noindex page 구분
   - 일정/날짜 관련 페이지에서 SEO 확장 여지 고려

4. **Adsense 친화 레이아웃 고려**
   - 광고를 과도하게 넣지 않되, 추후 삽입 가능한 레이아웃 구조 제안
   - CLS를 줄이기 위한 광고 영역 reserved space 고려
   - 콘텐츠/도구형 페이지에서 광고 배치 가능 포인트 설계
   - 정책 위반 가능성이 있는 UX는 피하기

5. **전환율/행동 유도 개선**
   - CTA 문구/배치 개선
   - 첫 방문자와 재방문자의 행동을 각각 고려
   - 회원가입/일정 생성/공유/참여 등의 목표 행동을 더 쉽게 유도

6. **실제 코드베이스 기준 실행 가능한 개선안 제시**
   - 단순 아이디어가 아니라 실제 리팩토링 가능한 수준으로 정리
   - 현재 코드 구조를 분석한 뒤, 어떤 컴포넌트/페이지/스타일 구조를 어떻게 바꿔야 하는지 제안
   - 가능하면 바로 적용 가능한 코드 수정안까지 포함

# 3. 반드시 고려할 제품 관점

리팩토링 시 아래 관점을 충분히 반영해주세요.

## 3-1. 사용자 유형
서비스에는 최소 다음 사용자가 있습니다.

- **첫 방문 사용자**
  - “이 서비스가 뭐 하는 건지” 빠르게 이해해야 함
  - 진입 장벽이 낮아야 함

- **실사용 사용자**
  - 빠르게 일정 확인/생성/조율/공유를 원함
  - 반복 동작이 편해야 함

- **검색 유입 사용자**
  - 특정 날짜, 일정, 캘린더 관련 정보/도구를 찾다가 들어올 수 있음
  - 랜딩 페이지 품질과 정보 전달력이 중요함

## 3-2. 서비스 컨셉
DateDate는 단순한 CRUD 캘린더가 아니라,
**“날짜를 중심으로 약속과 행동을 연결하는 도구”**
처럼 보이도록 UX를 정리해주세요.

즉 UI는 다음 인상을 주는 것이 좋습니다.

- 빠르다
- 직관적이다
- 일정 조율에 강하다
- 공유하기 쉽다
- 군더더기 없다
- 신뢰감 있다

## 3-3. 브랜드/톤
아래 톤을 선호합니다.

- 깔끔함
- 가벼운 친근감
- 지나치게 업무툴스럽지 않음
- 하지만 장난감 같지도 않음
- 미니멀, 선명한 위계, 적절한 강조

# 4. UX 관점에서 중점 점검 항목

코드베이스를 분석한 후, 아래 항목을 중심으로 진단해주세요.

## 4-1. 첫 화면 / 랜딩
- 첫 화면에서 서비스가 무엇인지 3초 안에 이해되는가?
- 핵심 CTA가 명확한가?
- “바로 시작”과 “먼저 이해”의 균형이 있는가?
- 모바일 첫 화면에서 핵심 기능이 너무 아래로 밀리지 않는가?

## 4-2. 내비게이션
- 상단 네비 / 하단 탭 / 사이드 구조가 역할에 맞는가?
- 메뉴명이 사용자 언어로 되어 있는가?
- 현재 위치와 다음 행동이 자연스럽게 연결되는가?

## 4-3. 캘린더/일정 인터랙션
- 날짜 선택, 일정 생성, 수정, 참여 흐름이 직관적인가?
- 클릭 가능한 요소와 정보 표시 요소가 잘 구분되는가?
- 리스트/캘린더/상세 보기 사이의 전환이 매끄러운가?

## 4-4. 일정 조율/공유 UX
- DateDate의 차별점이 드러나는가?
- 상대와 날짜를 맞추는 흐름이 쉽고 설득력 있는가?
- 링크 공유 / 참여 / 상태 확인의 흐름이 단순한가?

## 4-5. 피드백/상태표시
- 로딩, 빈 상태, 에러, 성공 상태가 충분히 설계되어 있는가?
- 사용자가 지금 어떤 상태인지 헷갈리지 않는가?

## 4-6. 접근성
- 버튼/링크/입력요소의 구분이 명확한가?
- 색상 대비, 포커스, 키보드 탐색, aria 구조를 개선할 수 있는가?

# 5. SEO 요구사항

다음 SEO 요구사항을 반드시 반영해주세요.

## 5-1. 기술 SEO
- semantic HTML 적극 사용
- heading 구조(h1-h2-h3) 정상화
- 중복 title/meta 방지
- canonical 전략 제안
- sitemap / robots 전략 제안
- structured data 적용 가능성 검토
  - WebSite
  - SoftwareApplication
  - FAQPage
  - BreadcrumbList
  - Event (필요 시)

## 5-2. 페이지 유형별 SEO 전략
페이지를 아래처럼 분류해서 각각 SEO 적합성을 평가해주세요.

- 홈/랜딩 페이지
- 기능 소개 페이지
- 일정 생성/참여 페이지
- 사용자 개인 페이지(보통 noindex 후보)
- 공유 가능한 공개 페이지
- 블로그/가이드/FAQ 성격의 콘텐츠 페이지(있다면 강화 포인트 제안)

## 5-3. 콘텐츠 SEO 확장성
DateDate가 단순 앱을 넘어서 검색 유입을 받으려면
어떤 유형의 페이지가 추가 가능할지 제안해주세요.

예:
- 날짜 계산/일정 조율 관련 도구형 페이지
- 약속 잡기 가이드
- 캘린더 활용 팁
- 특정 상황별 일정 조율 템플릿
- FAQ / 사용법 페이지

단, 본 서비스의 핵심 기능과 동떨어진 콘텐츠 농장은 지양해주세요.

# 6. Adsense 고려사항

Adsense 삽입을 전제로, 다음을 고려해주세요.

- 광고를 어디에 두면 UX를 해치지 않는가
- 모바일에서 광고가 핵심 인터랙션을 방해하지 않도록 구조화
- 광고 삽입용 슬롯을 어디에 reserved layout으로 둘 수 있는가
- 랜딩/콘텐츠/도구형 페이지별 광고 전략 차이 제안
- 일정 생성/참여와 같은 핵심 task flow에서는 광고 최소화 또는 제외
- 광고 때문에 신뢰감이 떨어지지 않도록 균형 유지

가능하면 다음 3개 레벨로 구분해주세요.

1. 광고 비노출 권장 영역
2. 제한적 광고 가능 영역
3. 콘텐츠형 페이지에서 광고 효율이 좋은 영역

# 9. 추가 제약

- 과도한 애니메이션 지양
- 광고 때문에 앱의 핵심 기능성이 손상되면 안 됨
- SEO를 위해 억지 텍스트를 늘리지 말 것
- 모바일 사용성을 최우선으로 보되 데스크탑도 자연스럽게
- 단순히 “예쁜 UI”보다 “행동이 쉬운 UI”를 우선
- 서비스 특성상 날짜/시간/일정 관련 정보 위계가 가장 중요
- 공개 페이지와 개인 페이지의 성격을 분리해서 설계
- 필요하면 noindex 전략을 적극 사용
- 접근성과 성능을 희생하지 말 것
```
